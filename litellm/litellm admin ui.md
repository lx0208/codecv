# LiteLLM Admin UI 

## 1. LiteLLM Proxyとは

LiteLLM Proxyは、複数のLLMを統一的に扱うための
 **LLMゲートウェイ（API統合基盤）**です。

主な機能：

- OpenAI互換APIの提供
- 複数モデルの統合（OpenAI / Vertex AI / vLLMなど）
- 認証・キー管理
- ログ管理
- 利用制御（制限・課金）



## 2. 事前準備

### 必須構成

- PostgreSQL（データ保存）
- LiteLLM Proxy（本体）



### 重要設定

#### ① master_key

- 管理用の最上位キー
- Key / User / Team の管理に必須
- 未設定だとUI操作不可

------

#### ② 環境変数

- DB接続情報
- Admin UIのログインID / パスワード

## 3. Admin UIの主な機能

------

### 3.1 Virtual Key（仮想キー）

- 実際のモデルキーではなく「アクセス用キー」
- 利用者ごとに発行可能

#### 命名ルール（重要）

```
team-userNo-yyyymmdd
```

例：

```
ST01-nts00000-20260319
```

👉 ログ分析のため必須

### 3.2 Team（チーム）

- 論理的なグループ管理単位

用途：

- 部門管理

👉 モデル・利用量の制御が可能



### 3.3 User（ユーザー）

- 利用者（人）を表す

推奨：

- 社員番号で管理

👉 UserとKeyを紐づけることで
 ログ追跡が可能になる

### 3.4 Log（ログ）

最重要機能

取得できる情報：

- user_id（誰が）
- key_name（どのキーで）
- model（どのモデル）
- token使用量
- コスト
- レイテンシ
- リクエスト / レスポンス内容

## 4. 運用でできること

- 利用者単位の追跡
- コスト分析
- 異常検知
- プロンプト検証
- パフォーマンス分析

## 5. ベストプラクティス

------

### ✅ Key命名ルールを統一する

→ ログ分析の精度向上

------

### ✅ 必ずUserを紐づける

→ 誰が使ったかを特定可能

------

### ✅ Teamで管理する

→ プロジェクト・環境ごとに分離

------

### ✅ ログを定期的に確認する

→ 運用・改善の最重要データ
